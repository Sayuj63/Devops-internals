Architecture

Six layers, written down honestly.

This brief explains every layer of the platform and the design tradeoffs behind it. The diagrams live at architecture.svg and deployment.svg.

1 · Client layer

The operator dashboard is hand-written HTML with Tailwind utility classes loaded from the CDN and a small vanilla-JS controller. There is no build step. Why: a console used by maybe twelve people does not need a framework, and a static asset has dramatically simpler deployment, security, and DR characteristics than a SPA fetched from a CDN bucket.

2 · Edge layer

AWS Application Load Balancer in three AZs, with ACM-issued TLS, WAF rules attached for OWASP Top 10, and Route 53 weighted records that target a DR backup in us-west-2. NLB was rejected because we need L7 path routing for the dashboard, API, and Grafana to share a hostname.

terraform/modules/alb/main.tf · excerpt
resource "aws_lb" "sim_ops" {
  name               = "sim-ops-alb"
  load_balancer_type = "application"
  subnets            = aws_subnet.public[*].id
  security_groups    = [aws_security_group.alb.id]
  drop_invalid_header_fields = true
  enable_deletion_protection = true
  tags = local.common_tags
}

3 · Compute layer · why Kubernetes, why distroless

EKS gives us multi-AZ scheduling, HPA, PDBs, and NetworkPolicies without us writing our own scheduler. Each workload runs from a distroless base image (gcr.io/distroless/python3-debian12) so the runtime has no shell, no apt, and a much smaller attack surface. Containers run as non-root (UID 65532), with read-only root filesystem and dropped capabilities.

We chose Kustomize over Helm for two reasons: this codebase has three environments, not three hundred installations, so values templating is overkill; and Kustomize overlays are diff-readable in a code review, where Helm's templating obscures what actually ships.

k8s/base/api/deployment.yaml · excerpt
apiVersion: apps/v1
kind: Deployment
spec:
  template:
    spec:
      securityContext: { runAsNonRoot: true, runAsUser: 65532, fsGroup: 65532 }
      containers:
      - name: api
        image: ghcr.io/itm-sayuj/sim-ops/api:v1.0.0
        resources:
          requests: { cpu: "100m", memory: "128Mi" }
          limits:   { cpu: "500m", memory: "512Mi" }
        livenessProbe:  { httpGet: { path: /healthz, port: 8080 }, periodSeconds: 10 }
        readinessProbe: { httpGet: { path: /ready,   port: 8080 }, periodSeconds: 5 }
        securityContext:
          allowPrivilegeEscalation: false
          readOnlyRootFilesystem: true
          capabilities: { drop: ["ALL"] }

4 · Data layer · why RDS Multi-AZ over Aurora

Aurora would give us a faster failover and decoupled storage scaling, at roughly 1.6× the cost. For a 100 GB telecom inventory with bursty writes during activation campaigns, RDS Postgres Multi-AZ hits the same RPO/RTO targets at a fraction of the bill. We get PITR (5-minute granularity), automated snapshots, and a cross-region read replica we promote during DR-02.

Connection management goes through pgbouncer in transaction-pooling mode so HPA-driven pod churn does not exhaust the 200-connection budget on the db.r6g.large primary.

5 · Secrets · Vault, not Secrets Manager

AWS Secrets Manager would have been simpler, but it does not issue short-lived dynamic credentials, and ops needed first-class lease revocation for DR-05 (compromised image). Vault's database secrets engine issues a fresh Postgres role per pod with a 1-hour lease, and the Vault Agent Injector renews it before expiry. The Vault control plane runs in its own namespace, sealed with Shamir + auto-unseal via AWS KMS.

6 · Observability · RED, USE, and one SLO

Prometheus scrapes API, worker, kube-state-metrics, and node-exporter. Recording rules pre-compute the SLI burn-rate so dashboards stay snappy. Grafana hosts four dashboards: SLO board, RED board, USE board, and capacity. ELK ships pod logs via Filebeat; Logstash enriches with iccid, trace_id, and tenant_id; Kibana saves searches the on-call uses during DR-01.

Exactly eight alerts. Anything we do not have a documented runbook for does not get to page humans.

Design rejections (documented for the brief)

  • Service mesh (Istio / Linkerd) — rejected. Eight services do not justify the complexity tax. mTLS happens at the ingress; pod-to-pod traffic is constrained by NetworkPolicy.
  • GitOps controller (ArgoCD / Flux) — deferred. Jenkins + kubectl apply is sufficient for two environments. ArgoCD becomes worthwhile around the time you have ten clusters.
  • Aurora Postgres — rejected on cost / benefit, see above.
  • Datadog / New Relic — rejected. The brief evaluates open-source observability; vendor SaaS would make the case study less defensible academically.
