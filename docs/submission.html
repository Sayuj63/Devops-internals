01 · Overview
Section 01

Project Overview

A telecom-operator-grade pipeline for activating SIM cards, modelled on real-world pain points and built with production patterns end-to-end.

The pain points

  • Slow activations — SIM activation often takes minutes per card; we target sub-second.
  • Inconsistent infra — every region runs a slightly different copy of the platform.
  • Zero visibility — when activations get stuck, operators don't know where.
  • Secrets sprawl — DB passwords, HLR keys and JWT material spread across config files.

What we built

A working FastAPI service backed by Postgres, modelling the real telecom state machine: PENDING → ALLOCATED → ACTIVE → SUSPENDED / PORTED / RECYCLED. Every transition writes an immutable audit event. Around it, the full DevOps lifecycle:

Containerization

Multi-stage Dockerfiles · non-root · tini PID 1 · compose-orchestrated local stack.

CI/CD

Jenkins pipeline — 9 stages with Trivy scan, smoke test and manual prod gate.

IaC

Terraform · VPC · EKS 1.29 · RDS Postgres 15 multi-AZ.

Kubernetes

Hardened pod-spec, HPA 3–12, PDB, NetworkPolicy, Vault agent injector.

Observability

Prometheus + Grafana + Alertmanager + ELK — RED metrics + business KPIs.

Secrets

HashiCorp Vault — kv/sim-prov tree with DB, HLR, JWT and audit-signing keys.

02 · Architecture
Section 02

Architecture & Data Flow

Operators hit the dashboard, which calls the FastAPI service. State lands in Postgres, metrics in Prometheus, logs in ELK, and secrets are pulled from Vault at startup. The async worker drives HLR provisioning — every step audited.

Frontend

Operator Console (Vite + vanilla JS)

Live KPI cards, donut by status, audit feed polling every second, inventory table with status pills, transition actions.

API

FastAPI · uvicorn (2 workers)

OpenAPI schema, async SQLAlchemy + asyncpg, prometheus_client metrics, structlog JSON logs with request_id propagation.

Worker

Async tick loop · 5s cadence

Polls ALLOCATED SIMs, calls mock-HLR for provisioning, transitions to ACTIVE on success — emits HLR call counters labelled by outcome.

State

Postgres 15 · Alembic-managed schema

Tables: plans, sims, msisdn_pool, audit_events. Indexed by status and last_transition_at for hot operator queries.

Observability

Prometheus + Grafana + Alertmanager + ELK (Filebeat → Logstash → Elasticsearch → Kibana)

RED + business KPIs; 8 alert rules; structured logs flowing live into sim-prov-* indices.

Secrets

HashiCorp Vault · kv-v2 mount at secret/sim-prov

DB creds, HLR API key, JWT signing keys and audit-signing material — pulled at app start, never written to disk.

04 · Containerization
Section 04

Containerization

Thirteen services orchestrated with docker compose: api, worker, postgres, mock-hlr, frontend, vault, prometheus, alertmanager, grafana, elasticsearch, logstash, kibana, filebeat.

Image patterns

  • Multi-stage builds — slim runtime; code change rebuilds in seconds via cached layers.
  • Non-root — dedicated app user, uid 1001, on every image.
  • tini as PID 1 — proper signal handling and zombie reaping.
  • curl healthchecks — every service has a meaningful /healthz probe.

05 · Metrics
Section 05

Observability — Metrics

RED metrics (rate, errors, duration) layered with business KPIs — every panel queries a metric the API actually exports, validated end-to-end.

The exported series

  • sim_state_transitions_total{from_status, to_status} — every lifecycle move.
  • sim_activation_latency_seconds — histogram → p50 / p95 / p99 via recording rules.
  • sim_count_by_status{status} — gauge refreshed every 10s by an async background task.
  • msisdn_pool_remaining, hlr_calls_total{outcome} — pool depth, HLR success rate.

08 · Secrets
Section 08

Secrets Management

HashiCorp Vault running in dev mode for the local stack, hardened K8s deployment pattern for production with the Vault Agent Injector rendering secrets at pod start.

What we store

  • secret/sim-prov/db — Postgres host, port, user, password, dbname.
  • secret/sim-prov/hlr — HLR mock endpoint and API key.
  • secret/sim-prov/jwt — signing key and algorithm for the operator console.
  • secret/sim-prov/audit-signing — HMAC key used to sign every audit-event row.

Read pattern

The API at startup makes one call to vault kv get secret/sim-prov/db, populates the SQLAlchemy DSN, then drops the token. Production uses the K8s Vault Agent Injector with auto-renewal and never writes secrets to disk.

Kubernetes · hardened pod spec

  • HPA 3–12 replicas at 70% CPU; PDB minAvailable: 2; NetworkPolicy restricting ingress and egress.
